# IITM Handbook RAG Chatbot

A from-scratch Retrieval-Augmented Generation (RAG) chatbot for the IITM BS Degree student handbook. No LangChain, no LlamaIndex вҖ” raw Python only.

**Stack:** Voyage AI (`voyage-3`) for embeddings В· Claude (`claude-haiku-4-5`) for reranking & generation В· numpy for cosine similarity В· FastAPI for the API В· plain JSON as the vector store.

---

## How to Run

```bash
# 1. Install dependencies
pip install -r requirements.txt

# 2. Set your Anthropic API key in .env
#   ANTHROPIC_API_KEY=sk-ant-...
# (Voyage AI uses the same key via voyageai.Client)

# 3. Build the vector store (run once)
python ingest.py

# 4. Start the API server
uvicorn main:app --reload
```

The API will be available at `http://localhost:8000`.

## File Overview

| File | Purpose |
|------|---------|
| `ingest.py` | One-time script: fetches the Google Doc, chunks it by headings, embeds all chunks via Voyage AI, saves `vector_store.json` |
| `retriever.py` | Core RAG logic: `load_store`, `cosine_similarity`, `retrieve`, `rerank`, `generate` вҖ” all model calls live here |
| `main.py` | FastAPI server: loads the store at startup via lifespan event, exposes `POST /chat` and `GET /health` |
| `vector_store.json` | Plain JSON vector store вҖ” generated by `ingest.py`, read-only at runtime |
| `requirements.txt` | Python dependencies |
| `.env` | Anthropic API key (never commit this) |

### Why each file is structured this way

- **`ingest.py` is separate** so embedding costs are paid once, not on every server restart. The JSON store is human-readable and trivially inspectable.
- **`retriever.py` has no FastAPI imports** вҖ” pure logic, easily testable in isolation (see `test_queries.py`).
- **`main.py` is thin** вҖ” only wires the HTTP layer to the retriever. All business logic stays in `retriever.py`.
- **No vector database** вҖ” for a single document the JSON file is fast enough. Linear scan over ~100 chunks with numpy takes <1 ms.

### Design decisions

- **Voyage AI embeddings** вҖ” Anthropic's recommended embedding partner. `voyage-3` uses separate `input_type="document"` vs `"query"` modes, which improves retrieval quality over symmetric embeddings.
- **Two-stage retrieval** (cosine similarity вҶ’ LLM reranking) gives better precision: the first stage is fast and cheap; the second uses language understanding to pick the most relevant passages.
- **cosine_similarity is manual** вҖ” implemented with `numpy.dot` / `numpy.linalg.norm`, no sklearn.
- **All model calls have `try/except`** with descriptive errors that propagate as HTTP 500 with context.
